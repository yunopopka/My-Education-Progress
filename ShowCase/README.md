Краткое ТЗ (Техническое задание):
Разработать архитектуру консольного симулятора «Система Планетарной Обороны». Требовалось применить продвинутые концепции C#: паттерны Singleton (Командный центр) и Object Pool (Пул лазерных лучей), систему наследования для врагов (Астероиды и Корабли), математические расчеты дистанции, а также фильтрацию объектов.

✅ Что сделано отлично (Strengths):

    Паттерн Singleton: Идеально реализован класс CommandCenter со скрытым конструктором и защищенной логикой изменения лимитов энергии.

    Object Pool (Оптимизация памяти): Успешно внедрен класс BeamPool<T> на базе коллекции Queue<T>, что предотвращает утечки памяти при массовой стрельбе.

    Безопасная модификация коллекций: Студент самостоятельно применил обратный цикл for (int i = list.Count - 1; i >= 0; i--) для удаления убитых врагов из списка EnemyGroup «на лету», избежав критической ошибки смещения индексов, на которой спотыкается большинство новичков.

    Игровой цикл (Game Loop): Написан полноценный интерактивный цикл с консольным меню и управлением ресурсами.

🔧 Зоны для рефакторинга (Где потребовались исправления):

    Нарушение инкапсуляции (Логика урона): Пушка (LaserGun) напрямую отнимала HP у врага (unit.HP -= EnergyShoot), минуя метод Damage(). Из-за этого враги не фиксировали свою смерть и не меняли статус IsAlive, продолжая существовать с отрицательным здоровьем. Урон должен наноситься только через вызов соответствующего метода юнита.

    Жесткая связность (Tight Coupling): При смерти враг напрямую обращался к синглтону CommandCenter.Center.AddScore(). В идеальной архитектуре враг должен вызывать событие Action OnDeath, а Штаб — подписываться на него (Паттерн Observer).

    Отсутствие гибкости в Радаре: Радар фильтровал цели через жесткие проверки if (count == 1/2). Для большей масштабируемости следовало применить делегат Func<EnemyUnit, bool>, передавая правила фильтрации извне (с помощью лямбда-выражений), чтобы Радар стал полностью универсальным.
